PROJECT OVERVIEW (LIVING DOCUMENT)
Last updated: 2026-03-30 (price-only + richer LightGBM focus)

==================================================
TECHNICAL TERMS
==================================================

1) Purpose
- Build a price-based market prediction pipeline.
- This repo focuses on price-only modeling.
- The active training path uses LightGBM.
- Logistic regression remains an internal benchmark only.
- Older news-related code remains in the codebase for reference, but it is not part of the active workflow.

2) Active Architecture
- Backend: FastAPI
- Frontend: React + Vite + Tailwind
- DB access: SQLAlchemy ORM
- DB target: PostgreSQL (SQLite fallback supported)
- Scheduling: APScheduler
- Active data source: Yahoo Finance via yfinance
- Active model family: LightGBM on tabular snapshot features

3) Active Data Flow
- Ingestion layer:
  - Yahoo OHLCV rows -> market_prices
  - every job run logged -> ingestion_runs
- Feature layer:
  - rolling aggregation by ticker/window -> feature_snapshots
  - active emphasis is on pure price features
  - feature set now includes richer structure signals:
    multi-horizon returns, EMA gaps, RSI, Bollinger z-score, breakout, drawdown,
    downside volatility, trend slopes, volume regime features, and new relative-strength features against SPY, QQQ, and sector ETFs
- Label layer:
  - future return + binary direction target -> market_labels
- Model layer:
  - join features + labels by ticker/date
  - time-ordered train/validation split
  - active model: LightGBM
  - benchmark model: scaled logistic regression
  - LightGBM candidate trial selection during training
  - walk-forward evaluation
  - threshold analysis for hold decisions
  - top feature importance capture
  - every retrain stored in model_runs
- Observability layer:
  - /data/status and /data/quality endpoints
  - prediction_logs table captures inference usage
- UI layer:
  - one-click dashboard for ingestion, feature runs, training, prediction, and model history comparison
  - model status emphasizes active model type, selected trial, logistic benchmark comparison, feature importance, threshold sweep, and experiment-matrix results

4) Active Tables
- market_prices
  - OHLCV rows per ticker/timestamp/interval
- feature_snapshots
  - model input rows built from price behavior
  - includes:
    price_close, return_1d, price_return_2d, price_return_3d, price_return_5d,
    price_return_10d, price_return_15d, price_return_20d,
    ma_gap_5d, ma_gap_20d, ma_crossover_5_20,
    ema_gap_12d, ema_gap_26d, ema_crossover_12_26,
    range_pct_1d, atr_14_pct, rolling_volatility_20d, downside_volatility_20d,
    volatility_regime_60d, volume_zscore_20d, volume_change_5d, volume_ratio_20d,
    rsi_5, rsi_14, bollinger_z_20, breakout_20d, drawdown_20d,
    trend_slope_10d, trend_slope_20d, up_day_ratio_10d
- market_labels
  - supervised targets from future returns
- model_runs
  - stored retrain history with metrics, feature set, horizon, threshold, recommended hold threshold, and full metrics_json
- prediction_logs
  - stored prediction calls/results
- ingestion_runs
  - pipeline audit log

7) Current State
- The repo behaves as a price-only project by default.
- LightGBM is the active model path.
- Logistic regression is retained only as a benchmark inside training metadata/history.
- Retrains are persisted and surfaced in the UI.
- The feature set is much richer than the earlier baseline.
- LightGBM candidate-trial selection now happens automatically during training.
- The latest run uses a 5-year, 20-ticker dataset and now shows LightGBM slightly beating the logistic benchmark on the main validation metrics.

8) Archived / Optional Work
- News ingestion, NLP, historical news import, and GDELT normalization still exist in the repo.
- They are not part of the active default workflow.
- Treat them as parked experimental infrastructure, not current product scope.

9) Transformer Status
- Transformer work is intentionally deferred.
- A transformer should not be bolted onto the current snapshot table design.
- If pursued later, it should use a sequence-style dataset builder built around rolling ticker windows.

10) Next Technical Milestones
- use the built-in experiment matrix to compare target horizons and thresholds using the richer feature set, with the active default now set to 5d / 0.004
- keep monitoring whether LightGBM can surpass the logistic benchmark
- tighten constrained strategy summaries toward more realistic backtesting if needed
- keep transformer work deferred until sequence data exists

==================================================
VERY SIMPLE TERMS
==================================================

What this project does now:
- it downloads stock price data
- it turns that into richer training rows
- it trains several LightGBM variants and picks the best one
- it also trains a logistic model in the background just to compare results
- it stores every retrain so you can see if the new model is actually better
- it shows the chosen tree settings, best feature drivers, and threshold sweep in the dashboard
- it gives predictions in the dashboard

What this project is not focused on now:
- it is not mainly a news project anymore
- it is not doing transformer work yet
- news tools are still in the repo, but they are parked

Simple current reality:
- the system is better built and better instrumented now
- the dashboard can now compare whole horizon/threshold grids without manual note-taking
- and the latest broader-data run now has LightGBM narrowly beating logistic on the main validation numbers
