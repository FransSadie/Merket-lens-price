PROJECT OVERVIEW (LIVING DOCUMENT)
Last updated: 2026-03-24 (price-only focus)

==================================================
TECHNICAL TERMS
==================================================

1) Purpose
- Build a price-based market prediction pipeline.
- This repo focuses on price-only modeling.
- Older news-related code remains in the codebase for reference, but it is not part of the active workflow.

2) Active Architecture
- Backend: FastAPI
- Frontend: React + Vite + Tailwind
- DB access: SQLAlchemy ORM
- DB target: PostgreSQL (SQLite fallback supported)
- Scheduling: APScheduler
- Active data source: Yahoo Finance via yfinance

3) Active Data Flow
- Ingestion layer:
  - Yahoo OHLCV rows -> market_prices
  - every job run logged -> ingestion_runs
- Feature layer:
  - rolling aggregation by ticker/window -> feature_snapshots
  - active emphasis is on pure price features
- Label layer:
  - future return + binary direction target -> market_labels
- Model layer:
  - join features + labels by ticker/date
  - time-ordered train/validation split
  - scaled logistic regression baseline
  - walk-forward evaluation
  - threshold analysis for hold decisions
  - every retrain stored in model_runs
- Observability layer:
  - /data/status and /data/quality endpoints
  - prediction_logs table captures inference usage
- UI layer:
  - one-click dashboard for ingestion, feature runs, training, prediction, and model history comparison

4) Active Tables
- market_prices
  - OHLCV rows per ticker/timestamp/interval
- feature_snapshots
  - model input rows built from price behavior
  - includes:
    price_close, return_1d, price_return_3d, price_return_5d, price_return_10d,
    price_return_20d, ma_gap_5d, ma_gap_20d, ma_crossover_5_20,
    range_pct_1d, atr_14_pct, rolling_volatility_20d, volatility_regime_60d,
    volume_zscore_20d, volume_change_5d
- market_labels
  - supervised targets from future returns
- model_runs
  - stored retrain history with metrics, feature set, horizon, threshold, and recommended hold threshold
- prediction_logs
  - stored prediction calls/results
- ingestion_runs
  - pipeline audit log

5) Active Modules
- app/ingestion/market_data.py
  - market price download + normalization
- app/ingestion/pipeline.py
  - active market ingestion flow
- app/features/pipeline.py
  - feature aggregation + label generation with multi-horizon price features
- app/models/dataset.py
  - training dataframe builder + latest feature lookup
- app/models/train_baseline.py
  - active model training, metric generation, threshold analysis, model-run persistence
- app/models/inference.py
  - prediction from latest active model
- app/api/routes.py
  - API endpoints for the active workflow and model history
- frontend/src/App.jsx
  - dashboard UI with model-run history comparison

6) Active API Endpoints
- GET /health
- POST /ingest/run
- GET /ingest/status
- POST /pipeline/run
- GET /pipeline/status
- POST /model/train
- GET /model/status
- GET /model/history?limit=20
- GET /predict?ticker=SPY
- GET /prediction/logs?limit=100
- GET /data/status
- GET /data/quality
- POST /run/full?train_model=true
- GET /docs/text

7) How to Run
- Ingestion only:
  - python -m app.ingestion.run_once
- Features + labels:
  - python -m app.features.run_once
- Train model:
  - python -m app.models.train_baseline
- API server:
  - python -m uvicorn app.main:app --reload
- Frontend:
  - cd frontend
  - npm install
  - npm run dev

8) Current State
- The repo now behaves as a price-only project by default.
- Price-only training is the active model path.
- Retrains are persisted and surfaced in the UI.
- The model now uses richer price features and configurable horizon/threshold settings.

9) Archived / Optional Work
- News ingestion, NLP, historical news import, and GDELT normalization still exist in the repo.
- They are not part of the active default workflow.
- Treat them as parked experimental infrastructure, not current product scope.

10) Next Technical Milestones
- tune the current price-only feature set
- compare 1d/3d/5d targets and different return thresholds
- add lightweight backtesting / paper-trade evaluation
- keep price-only as the benchmark
